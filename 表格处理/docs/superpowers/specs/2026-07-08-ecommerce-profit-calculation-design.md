# 电商多平台利润表自动计算系统 - 设计文档

## 1. 项目概述

### 1.1 目标
构建一个专业级前端后台系统，支持拼多多、淘宝、抖音三大电商平台。用户上传平台原始导出表格和利润表模板，通过前端可视化配置计算规则，系统自动完成数据清洗、跨表关联、聚合计算，最终批量生成各店铺的利润表。

### 1.2 核心用户场景
1. 运营人员每月从各平台后台导出原始数据（订单、流水、账单等）
2. 上传利润表模板和原始数据到系统
3. 系统按预配置的规则自动计算利润、成本、扣点等关键指标
4. 批量导出填充好的利润表 Excel 文件

### 1.3 设计原则
- **平台级规则复用**：同平台所有店铺共用一套计算模型
- **全局外部数据**：商品成本、编码映射等维度表全平台通用
- **数据先行**：配置规则前需上传样本数据，确保列名可选
- **可视化配置**：所有计算规则通过下拉选择 + 简单输入完成，零代码
- **本地优先**：纯前端实现，数据不上传服务器，保障财务隐私

## 2. 核心概念

### 2.1 术语表

| 术语 | 定义 |
|------|------|
| **平台** | 拼多多 / 淘宝 / 抖音，每个平台有独立的模板和计算模型 |
| **店铺** | 隶属于某个平台的具体店铺，只存名称，不存独立规则 |
| **原始表格类型** | 平台导出的标准表格，如"订单导出"、"流水明细"、"账单详情" |
| **外部数据表** | 全局通用的维度表，如"产品成本维护表"、"商品编码映射表" |
| **利润表模板** | 每个平台唯一的 Excel 模板，包含待填充字段 |
| **计算模型** | 平台级的完整规则集，定义每个利润表字段如何计算 |
| **虚拟字段** | 在原始表基础上通过解析/转换生成的中间字段，供规则引用 |
| **样本文件** | 配置规则前上传的代表性文件，用于系统解析列结构 |

### 2.2 核心流程

```
创建平台（拼多多/淘宝/抖音）
    ↓
定义平台原始表格类型 + 上传利润表模板
    ↓
上传各表格类型的样本文件
    ↓
配置计算模型（字段映射、公式、关联、预处理）
    ↓
保存模型
    ↓
批量计算：选择店铺 → 上传实际数据 → 执行计算 → 预览 → 导出
```

## 3. 系统架构

### 3.1 模块划分

| 模块 | 职责 |
|------|------|
| **平台管理** | 增删改平台、管理店铺、查看模板和模型配置状态 |
| **模板中心** | 上传/管理利润表模板，识别待填充字段 |
| **计算模型** | 核心模块。为模板每个字段配置数据来源、预处理、计算公式 |
| **全局外部数据** | 管理全平台通用的成本表、编码映射表等维度表 |
| **批量计算** | 选择店铺、上传数据、执行计算、预览结果、批量导出 |
| **计算引擎** | 规则解析、AST 执行、跨表 JOIN、聚合运算、模板填充 |
| **数据预处理** | 数据清洗、过滤、虚拟字段生成、文本解析 |

### 3.2 技术栈

- **框架**：React 18 + TypeScript
- **构建工具**：Vite
- **UI 组件**：Tailwind CSS + shadcn/ui
- **状态管理**：Zustand
- **Excel 解析**：SheetJS (xlsx) 流式读取
- **计算引擎**：自研前端规则引擎（词法分析 → AST → 执行器）
- **数据持久化**：IndexedDB（缓存原始数据、计算结果、配置）
- **并发处理**：Web Worker（大数据量解析和计算不阻塞 UI）

### 3.3 数据流

```
用户操作（React UI）
    ↓
Zustand Store（平台/店铺/模板/规则/任务状态）
    ↓
Service 层：
  - ExcelService：解析/序列化 xlsx/csv
  - PreprocessService：数据清洗、虚拟字段生成
  - RuleEngine：公式词法分析、AST 构建、执行
  - TemplateService：模板字段识别、结果填充
  - BatchService：批量任务调度、Web Worker 分发
    ↓
Storage 层：IndexedDB（原始数据表、计算结果、配置快照）
```

## 4. 数据模型

### 4.1 平台（Platform）

```typescript
interface Platform {
  id: string;                    // 唯一标识
  name: '拼多多' | '淘宝' | '抖音';
  createdAt: number;
  // 关联
  shopIds: string[];             // 下属店铺
  templateId: string | null;     // 利润表模板
  calculationModelId: string | null; // 计算模型
  tableTypeIds: string[];        // 该平台涉及的原始表格类型
}
```

### 4.2 店铺（Shop）

```typescript
interface Shop {
  id: string;
  platformId: string;            // 所属平台
  name: string;                  // 店铺名称
  note?: string;
}
```

### 4.3 原始表格类型（RawTableType）

```typescript
interface RawTableType {
  id: string;
  platformId: string;            // 所属平台（外部数据表此字段为 null，表示全局通用）
  name: string;                  // 如"订单导出"、"流水明细"、"产品成本维护表"
  isExternal: boolean;           // 是否为全局外部数据表
  sampleColumns: string[];       // 从样本文件解析出的列名
  sampleFileName: string;        // 样本文件名（仅记录）
}
```

### 4.4 利润表模板（ProfitTemplate）

```typescript
interface ProfitTemplate {
  id: string;
  platformId: string;
  fileData: ArrayBuffer;         // 模板文件二进制
  fields: TemplateField[];       // 识别出的待填充字段
}

interface TemplateField {
  id: string;
  name: string;                  // 如"销售收入"、"平台扣点"、"净利润"
  cellAddress: string;           // 模板中的单元格位置，如"B5"
  dataType: 'number' | 'string' | 'date';
}
```

### 4.5 计算模型（CalculationModel）

```typescript
interface CalculationModel {
  id: string;
  platformId: string;
  fieldRules: FieldRule[];       // 每个模板字段的计算规则
  preprocessRules: PreprocessRule[]; // 数据预处理规则（全局，按表执行）
  virtualFields: VirtualField[]; // 虚拟字段定义
}

interface FieldRule {
  templateFieldId: string;       // 对应利润表字段
  sourceTableTypeId: string;     // 主要来源表
  sourceColumn: string;          // 来源列（可选，直接引用时）
  formula: FormulaNode | null;   // 公式 AST（复杂计算时）
  joinConfig?: JoinConfig;       // 跨表关联配置
  filterConfig?: FilterConfig;   // 过滤条件
  aggregationConfig?: AggregationConfig; // 聚合配置
}

interface JoinConfig {
  targetTableTypeId: string;     // 目标关联表
  sourceKey: string;             // 当前表关联键
  targetKey: string;             // 目标表关联键
  joinType: 'left' | 'inner';
}

interface FilterConfig {
  conditions: FilterCondition[];
  logic: 'AND' | 'OR';
}

interface FilterCondition {
  column: string;
  operator: '==' | '!=' | '>' | '<' | '>=' | '<=' | 'contains' | 'in';
  value: string | number | string[];
}

interface AggregationConfig {
  groupByColumns: string[];      // 分组维度
  aggregation: 'sum' | 'avg' | 'count' | 'min' | 'max';
  targetColumn: string;          // 聚合目标列
}

interface VirtualField {
  id: string;
  tableTypeId: string;           // 基于哪张表生成
  name: string;                  // 虚拟字段名
  generateRule: 'regex_extract' | 'formula' | 'map_replace' | 'date_extract';
  config: Record<string, any>;   // 规则参数
}

interface PreprocessRule {
  tableTypeId: string;
  type: 'filter' | 'deduplicate' | 'fill_na' | 'type_cast';
  config: Record<string, any>;
}
```

## 5. 计算引擎设计

### 5.1 公式语法

前端可视化配置最终生成 AST，但底层语法对齐 Excel 习惯：

| 功能 | 示例 |
|------|------|
| 基础运算 | `商家实收金额 - 邮费` |
| 跨表引用 | `LOOKUP(商品id, 产品成本维护表.成本单价)` |
| 条件判断 | `IF(售后状态 == "退款成功", 0, 用户实付金额)` |
| 聚合函数 | `SUM(商家实收金额)` |
| 条件聚合 | `SUMIF(售后状态, "已收货", 商家实收金额)` |
| 文本提取 | `REGEX_EXTRACT(商品规格, "(\d+)条装")` |
| 数值计算 | `ROUND(数量 * 单价, 2)` |

### 5.2 执行顺序

```
输入：店铺实际数据文件 + 计算模型

Step 1: 数据加载
  - 解析所有上传的 Excel/CSV 文件为内存数据表（JSON 数组格式）
  - 按 tableTypeId 分组存储

Step 2: 数据预处理（按表执行）
  - 应用过滤规则（如排除退款成功订单）
  - 去重
  - 空值填充
  - 类型转换

Step 3: 虚拟字段生成
  - 遍历 virtualFields 定义
  - 基于现有列生成新列
  - 示例：从"商品规格"提取数量 → 生成"解析数量"列

Step 4: 利润表字段计算（按字段顺序）
  For each fieldRule in fieldRules:
    a. 确定主数据表
    b. 如存在 joinConfig，执行关联合并（LEFT JOIN / INNER JOIN）
    c. 如存在 filterConfig，应用过滤条件
    d. 如存在 aggregationConfig，按维度分组聚合
    e. 如存在 formula，执行公式计算
    f. 如直接引用列，取对应值
    g. 将结果写入利润表模板的 cellAddress

Step 5: 模板导出
  - 将计算结果填充到模板副本
  - 生成新的 Excel 文件
```

### 5.3 精度控制

- 所有金额运算以**分**为单位转换为整数计算
- 最终结果除以 100 并保留 2 位小数
- 彻底避免 JavaScript 浮点精度问题

### 5.4 错误处理

- **字段级错误隔离**：某个字段计算失败不影响其他字段
- **错误精确定位**："净利润计算失败 → 原因：LOOKUP 关联键 '商品id' = 894186271413 在产品成本维护表中未找到匹配"
- **空值策略**：每个字段可独立配置空值处理方式（视为 0 / 跳过 / 报错）

## 6. 数据预处理与虚拟字段（关键补充）

### 6.1 为什么需要预处理层

实际平台导出的原始数据往往不能直接计算，需要清洗：
- 退款成功订单应排除
- 商品规格字段包含的"6条装"需要解析为数字 6
- 日期格式不统一
- 部分列包含中文"是/否"需要转为布尔值

### 6.2 预处理规则类型

| 类型 | 作用 | 示例 |
|------|------|------|
| **filter** | 过滤行 | `售后状态 != "退款成功"` |
| **deduplicate** | 去重 | 按订单号去重 |
| **fill_na** | 空值填充 | 空邮费填充为 0 |
| **type_cast** | 类型转换 | 将字符串金额转为数值 |

### 6.3 虚拟字段规则类型

| 类型 | 作用 | 示例 |
|------|------|------|
| **regex_extract** | 正则提取文本中的数字/模式 | 从`6条装 xxx`提取`6` |
| **formula** | 基于现有列计算新列 | `商品总价 / 商品数量` |
| **map_replace** | 映射替换 | `是→1, 否→0` |
| **date_extract** | 提取日期部分 | 从`2026-04-30 23:57:29`提取`2026-04` |

### 6.4 用户配置界面

在"计算模型"模块中，增加**"数据预处理"**标签页：
- 左栏：该平台的原始表格类型列表
- 右栏：当前选中表的预处理规则链（可拖拽排序）
- 虚拟字段作为预处理规则链的最后一步

## 7. 全局外部数据表设计

### 7.1 定位

外部数据表（产品成本维护表、商品编码映射表）**不隶属于任何平台**，全平台通用。

### 7.2 使用方式

1. 在"全局外部数据"模块上传样本并管理
2. 在任何平台的计算模型中，都可以选择外部数据表作为 JOIN 目标
3. 批量计算时，外部数据表与平台数据一起上传（或复用已有版本）
4. 支持版本覆盖：上传同名新文件自动替换旧数据

### 7.3 关联示例

```
平台：拼多多
主表：订单导出
外部表：产品成本维护表
关联键：订单导出.商品id = 产品成本维护表.商品ID
引用列：产品成本维护表.成本单价
虚拟字段：订单导出.解析数量 = REGEX_EXTRACT(商品规格, "(\d+)条装")
计算规则：成本 = LOOKUP(商品id, 产品成本维护表.成本单价) * 解析数量
```

## 9. 批量计算流程

```
用户选择平台 → 勾选店铺（1-N个）
    ↓
系统展示该平台所需的所有原始表格类型
    ↓
用户按表格类型上传文件（支持拖拽）
    ↓
【可选】上传/更新全局外部数据表
    ↓
用户点击"开始计算"
    ↓
系统为每个店铺创建一个计算任务
    ↓
Web Worker 池并行执行（最多同时 4 个）
    ↓
每个店铺的执行流程：
  1. 解析所有上传文件
  2. 执行预处理规则链
  3. 生成虚拟字段
  4. 按字段规则逐个计算
  5. 填充利润表模板
  6. 生成 Excel Blob
    ↓
所有店铺计算完成后
    ↓
结果预览表格展示关键指标
    ↓
用户点击"批量导出" → 打包下载所有店铺利润表
```

## 10. 利润表字段分层处理

### 10.1 问题背景

利润表各字段的数据来源复杂度不同：
- "销售收入"可能只需从订单表 SUM 一列，无需清洗
- "成本"需要跨表 JOIN + 解析商品规格数量 + 计算公式
- 如果所有字段都展示完整配置面板，界面臃肿且操作繁琐

### 10.2 解决方案：规则分级，按需展开

| 级别 | 处理需求 | 配置项展开 |
|------|----------|------------|
| **Level 0 - 直接引用** | 数据已存在，无需处理 | 仅展示：来源表 + 来源列 + 聚合方式（可选） |
| **Level 1 - 简单清洗** | 需过滤/排除某些行 | 展开：过滤条件构建器（AND/OR 多条件） |
| **Level 2 - 复杂计算** | 跨表 JOIN、虚拟字段、公式 | 展开全部：关联配置 + 虚拟字段选择 + 公式构建器 |

### 10.3 UI 实现

每个利润表字段配置面板顶部设置**"处理级别"下拉框**：

```
处理级别：○ 直接引用（默认）
          ○ 简单清洗
          ○ 复杂计算
```

选择后，下方配置项动态展开/收起，降低认知负担。

## 11. 商品规格解析规则库

### 11.1 问题背景

商品规格字段格式多样，无法用单一正则匹配：
- `"6条装 粉色+黄色+蓝色"` → 数量 = 6
- `"粉色+黄色+蓝色+绿色"` → 数量 = 4（"+"分隔计算）
- `"10个 xxx"` → 数量 = 10
- `"套装×3"` → 数量 = 3
- `"A/B/C"` → 数量 = 3

### 11.2 内置规则库

| 规则ID | 适用模式 | 正则表达式 | 示例 → 解析结果 |
|--------|----------|------------|-----------------|
| `qty_n条装` | "N条装"格式 | `(\d+)\s*条装` | `"6条装 xxx"` → `6` |
| `qty_n条` | "N条"格式 | `(\d+)\s*条` | `"5条 xxx"` → `5` |
| `qty_数字开头` | 开头数字 | `^(\d+)` | `"10个 xxx"` → `10` |
| `count_plus` | "+"号分隔数量 | 计算`+`出现次数 + 1 | `"粉色+黄色+蓝色"` → `3` |
| `qty_×n` | "×N"/"xN"格式 | `[×xX](\d+)` | `"套装×3"` → `3` |
| `qty_set` | 套装/件数格式 | `(\d+)\s*(?:件|套|个|支)` | `"3件套"` → `3` |
| `qty_slash` | "/"分隔数量 | 计算`/`出现次数 + 1 | `"A/B/C"` → `3` |
| `qty_parentheses` | 括号内数字 | `\((\d+)\)` | `"商品(6)"` → `6` |

### 11.3 用户配置流程

```
创建虚拟字段 → 选择"商品规格"列
    ↓
从下拉框选择"解析规则"（内置规则库列表）
    ↓
点击"预览解析"按钮
    ↓
系统用样本数据展示：
  - 解析成功率（如 95%）
  - 失败样本列表（如 5 条无法解析的规格）
    ↓
如内置规则不适用 → 点击"自定义规则"
    ↓
输入自定义正则表达式 → 测试 → 保存
```

### 11.4 规则更新机制

- 计算完成后，系统统计"解析失败率"
- 若失败率 > 10%，在结果页提示"建议新增解析规则"并展示失败样本
- 用户可随时编辑计算模型中的虚拟字段规则，无需重新配置整个模型

### 11.5 数据结构

```typescript
interface QuantityParseRule {
  id: string;
  name: string;                  // 规则名称
  category: 'built-in' | 'custom'; // 内置 or 用户自定义
  regex: string;                 // 正则表达式
  extractGroup: number;          // 提取第几个捕获组（默认 1）
  fallbackRule?: string;         // 失败时的备用规则ID
  enabled: boolean;
}
```

## 12. 批量上传智能分发

### 12.1 问题背景

实际场景：30 个店铺 × 4 种表格 = 120 个文件，手动为每个店铺分配每种表格极为繁琐

### 12.2 解决方案：统一上传 + 智能识别 + 预览调整

**核心流程**：

```
用户拖入所有文件到统一上传区
    ↓
系统逐个解析文件：
  1. 文件名提取店铺标识
     - 关键词匹配：用户预配置的店铺名列表
     - 正则模式：如"店铺A_订单.xlsx" → 店铺名="店铺A"
  2. 文件名提取表格类型
     - 关键词：订单、流水、账单、明细、成本、编码
     - 用户可自定义关键词映射
  3. 文件内容扫描（如有店铺名列）
     - 读取第一行数据，查找"店铺名称"字段
    ↓
自动分类分发：店铺 + 表格类型
    ↓
展示分发预览面板：
  - 左栏：店铺列表（展开显示每个店铺已分配的文件）
  - 右栏：未识别文件列表（用户手动拖拽分配或删除）
    ↓
用户确认/调整 → 点击"确认分发"
    ↓
开始计算
```

### 12.3 文件名识别策略

**店铺识别优先级**：
1. 用户预配置店铺名关键词（如 ["店铺A", "旗舰店", "专营店"]）
2. 文件名中出现的店铺标识（如 `_店铺A_`）
3. 文件内容第一行的店铺字段值

**表格类型识别关键词库**：

| 类型 | 关键词 |
|------|--------|
| 订单导出 | 订单、order |
| 流水明细 | 流水、账单、明细、bill |
| 账单详情 | 账单、结算 |
| 成本表 | 成本、成本维护 |
| 编码映射 | 编码、映射、商品ID |

### 12.4 额外优化

- **按店铺打包上传**：支持上传 `店铺A.zip`，内含该店铺的所有表格
- **上传历史记忆**：记住用户常用文件名格式，下次自动应用相同识别规则
- **智能合并**：同一店铺同类型多文件自动合并（如"店铺A_订单_1.xlsx" + "店铺A_订单_2.xlsx"）

### 12.5 UI 设计

**批量计算页面调整**：

```
顶部：平台选择 + 店铺勾选树
    ↓
中部左侧：【统一批量上传区】
  - 大尺寸拖拽区域
  - 提示："拖入所有店铺的所有表格文件"
  - 支持文件夹、zip 包
    ↓
中部右侧：【智能分发预览面板】
  - 三列：
    | 店铺列表（勾选状态） | 已分配文件（可展开查看） | 未识别文件 |
  - 操作：拖拽调整、删除、手动分配
    ↓
底部：【外部数据表区】（可折叠，默认复用已有）
    ↓
底部：【计算控制】
  - 确认分发按钮 → 开始计算按钮
  - 进度条 + 结果预览
```

## 13. 边界情况与容错

| 场景 | 处理策略 |
|------|----------|
| 上传的 Excel 列名与样本不一致 | 计算时提示"列名不匹配"，建议用户检查文件或更新样本 |
| 外部数据表缺少部分商品的成本 | 空值按该字段配置的策略处理（默认视为0并记录警告） |
| 商品规格无法解析出数量 | 虚拟字段生成失败，该记录标记为错误，不影响其他记录 |
| 某店铺缺少某个原始表格 | 计算前校验，提示必须上传的表格类型 |
| 浏览器内存不足（超大文件） | 流式解析 + 分片计算，单文件超过 50MB 时提示分批处理 |
| 计算过程中用户关闭页面 | IndexedDB 自动保存计算进度，重新打开可恢复（如设计支持） |
